## 🔴 Issue #1: Incorrect Weekly Date Ranges

### **Problem**
When selecting Q1 2026, weeks 1 and 2 were showing the same date range. The drill-down tooltip was displaying incorrect dates for weekly data points.

**User Report**:
> "Jan 2026 first week should include (Dec 29 - Jan 4) then second week and so on. Currently week 01 and week 02 showing same date range."

### **Root Cause**
1. **ISO Week Calculation Bug**: The code at line 1107-1121 was using a simplified week calculation that didn't account for ISO 8601 week standards.

```javascript
// ❌ OLD CODE (INCORRECT)
const simple = new Date(parseInt(year), 0, 1 + (weekNum - 1) * 7);
// This just adds 7 days per week, ignoring ISO week rules
```

2. **Hardcoded Week Dates Were Wrong**: The hardcoded week definitions (lines 340-370) didn't match ISO week standards.

```javascript
// ❌ OLD CODE (INCORRECT)
case "Q1_26_W1":
  return {
    start: new Date("2026-01-06"), // Wrong! Should start Dec 29
    end: new Date("2026-01-12T23:59:59Z"),
  };
```

### **Solution**

#### **Fix 1: Proper ISO Week Calculation ([server.js:1107-1137](backend/server.js#L1107))**

```javascript
// ✅ NEW CODE (CORRECT)
if (date.includes("W")) {
  const [year, weekPart] = date.split("-W");
  const weekNum = parseInt(weekPart);

  // ISO week calculation: Week 1 is the first week with 4+ days in the new year
  const jan1 = new Date(parseInt(year), 0, 1);
  const jan1Day = jan1.getDay(); // 0=Sunday, 1=Monday, ..., 6=Saturday

  // Calculate the Monday of week 1
  let week1Monday;
  if (jan1Day === 0) { // Sunday
    week1Monday = new Date(jan1);
    week1Monday.setDate(jan1.getDate() + 1); // Next Monday
  } else if (jan1Day <= 4) { // Monday-Thursday
    week1Monday = new Date(jan1);
    week1Monday.setDate(jan1.getDate() - (jan1Day - 1)); // Go back to Monday
  } else { // Friday-Saturday
    week1Monday = new Date(jan1);
    week1Monday.setDate(jan1.getDate() + (8 - jan1Day)); // Next Monday
  }

  // Calculate the start of the requested week
  startOfDay = new Date(week1Monday);
  startOfDay.setDate(week1Monday.getDate() + (weekNum - 1) * 7);
  startOfDay.setHours(0, 0, 0, 0);

  // End of week is Sunday
  endOfDay = new Date(startOfDay);
  endOfDay.setDate(startOfDay.getDate() + 6);
  endOfDay.setHours(23, 59, 59, 999);
}
```

**How it works**:
1. Find January 1 of the year
2. Check what day of the week it is
3. Calculate the Monday of ISO week 1 (first week with 4+ days in January)
4. Add (weekNum - 1) × 7 days to get the requested week

## 🔴 Issue #2: FRR Graph Showing Flatline

### **Problem**
The FRR (First Response Rate) metric card showed correct data, but the expanded chart view displayed a flatline (no variation).

**User Report**:
> "There is no graph I can see for the FRR. Each card has a graph but FRR has flatline. When I open expanded view I do get the results."

### **Root Cause**
The trends array was returning `frrMet` (count of tickets meeting FRR) but not `frrPercent` (the percentage value needed for the chart).

```javascript
// ❌ OLD CODE (MISSING frrPercent)
trends: trends.map((t) => ({
  date: t._id,
  solved: t.solved,
  frrMet: t.frrMet || 0, // ← Just the count, not percentage!
  // Missing: frrPercent
}))
```

**Why this caused a flatline**:
- The chart expects percentage values (0-100)
- It was receiving counts (e.g., 5, 10, 15) which are too small to show variation
- The overall FRR percentage was calculated at the stats level, but not per data point

### **Solution**

#### **Fix 1: Calculate FRR Percentage in Analytics Aggregation ([server.js:741-758](backend/server.js#L741))**

```javascript
// ✅ Add frrTotal to aggregation
const trends = await AnalyticsTicket.aggregate([
  { $match: matchConditions },
  {
    $group: {
      _id: { $dateToString: { format: dateFormat, date: "$closed_date" } },
      solved: { $sum: 1 },
      // ... other fields
      frrMet: { $sum: { $cond: [{ $eq: ["$frr", 1] }, 1, 0] } },
      frrTotal: { $sum: 1 }, // ✅ Add total count for percentage calculation
    },
  },
]);

// ✅ Calculate FRR percentage for each trend data point
const trendsWithFRRPercent = trends.map((t) => ({
  _id: t._id,
  solved: t.solved,
  avgRWT: t.avgRWT,
  avgFRT: t.avgFRT,
  avgIterations: t.avgIterations,
  positiveCSAT: t.positiveCSAT,
  frrMet: t.frrMet,
  frrPercent: t.frrTotal > 0 ? Math.round((t.frrMet / t.frrTotal) * 100) : 0, // ✅ FRR as percentage
}));
```

#### **Fix 2: Include frrPercent in Response ([server.js:919-933](backend/server.js#L919))**

```javascript
// ✅ Use trendsWithFRRPercent instead of trends
trends: trendsWithFRRPercent.map((t) => {
  const backlog = backlogCleared.find((b) => b._id === t._id);
  return {
    date: t._id,
    solved: t.solved,
    avgRWT: t.avgRWT ? Number(t.avgRWT.toFixed(2)) : 0,
    avgFRT: t.avgFRT ? Number(t.avgFRT.toFixed(2)) : 0,
    avgIterations: t.avgIterations ? Number(t.avgIterations.toFixed(1)) : 0,
    backlogCleared: backlog?.count || 0,
    positiveCSAT: t.positiveCSAT,
    frrMet: t.frrMet || 0,
    frrPercent: t.frrPercent || 0, // ✅ Include FRR percentage for graph
  };
}),
```

#### **Fix 3: Also Fixed in Live Stats Endpoint ([server.js:512-549](backend/server.js#L512))**

```javascript
// ✅ Add frrTotal tracking
const trendsMap = {};
data.dailyData.forEach((t) => {

    avgFRT: day.countFRT ? day.sumFRT / day.countFRT : 0,
```
